title: 'Type declarations moved to a types submodule'
description: 'HttpMethod, FunctionInfo, TriggerInfo, WorkerInfo, WorkerMetadata, IIIConnectionState (Rust) and MessageType (Python, Node) are no longer exported from the package root. Import them from the SDK types submodule instead. IIIConnectionState is now also reachable from the types submodule in Python and Node.'
---

## What changed

The following type/data declarations are no longer exported from the package root. They now live
in the SDK's `types` submodule. The root keeps exporting the runtime API (`register_worker` /
`registerWorker`, the client handle, triggers, channels, errors) unchanged.

| Symbol | SDK(s) |
| --- | --- |
| `HttpMethod` | Rust |
| `FunctionInfo`, `TriggerInfo`, `WorkerInfo`, `WorkerMetadata` | Rust |
| `IIIConnectionState` | Rust (moved); Python, Node (newly reachable from `types`) |
| `MessageType` | Python, Node |

## Migrate

### Rust

```rust
// Before
use iii_sdk::{HttpMethod, FunctionInfo, WorkerInfo, WorkerMetadata, IIIConnectionState};

// After
use iii_sdk::types::{HttpMethod, FunctionInfo, WorkerInfo, WorkerMetadata, IIIConnectionState};
```

`TriggerInfo` moves the same way (`iii_sdk::types::TriggerInfo`). Symbols that stayed at the crate
root — `register_worker`, `III`, `InitOptions`, `RegisterFunction`, `TriggerAction`, and the
others — are unaffected.

### Python

```python
# Before
from iii import MessageType

# After
from iii.types import MessageType
```

`IIIConnectionState` is now re-exported from the same submodule:

```python
from iii.types import IIIConnectionState
```

### Node

`MessageType` and `IIIConnectionState` are type-only exports on the new `iii-sdk/types` subpath:

```ts
// Before
import { MessageType } from 'iii-sdk'

// After
import type { MessageType, IIIConnectionState } from 'iii-sdk/types'
```

## Why

The package root now exposes runtime/behavior only; type and data declarations live in `types`.
This keeps the import surface for "what you call" separate from "what you pass and receive".
